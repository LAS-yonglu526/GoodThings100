## 1. 项目愿景与架构

### 项目定位
《好事100》(GoodThings100) 是一款基于 React Native + Expo 的精美数字清单 App。核心理念是用**果冻毛玻璃**视觉语言记录和追踪日常好事。

### 技术栈
| 层级 | 技术 |
|------|------|
| 框架 | React Native 0.81 + Expo Managed Workflow (SDK 54) |
| UI 组件 | 纯 React Native + `expo-blur` (iOS 原生毛玻璃) |
| 动画引擎 | `Animated.Value/ValueXY` + `PanResponder` + `LayoutAnimation` |
| 数据库 | SQLite (`expo-sqlite`, 单文件 `goodthings.db`) |
| 触觉反馈 | `expo-haptics` |
| 图片 | `expo-image-picker` + `expo-file-system/legacy` + `expo-image-manipulator` |
| 生物识别 | `expo-local-authentication` (Face ID) |
| 安全存储 | `expo-secure-store` (账号列表/密码标记/session token) |
| 分享截图 | `react-native-view-shot` + `expo-sharing` |
| 后端 | Supabase (Email OTP + 密码登录) |

## 2. 绝对红线与开发禁忌

### 2.1 动画引擎底线
```
🚫 绝对禁止 react-native-reanimated
   原因: iOS Hermes 不支持 # 私有属性语法 → 红屏 NativeWorklets
```

### 2.2 useNativeDriver 防报错准则
| 动画属性 | useNativeDriver |
|----------|----------------|
| `transform`, `opacity` | **必须 `true`** |
| `top/left/width/height` | **必须 `false`** |

同一个 `Animated.Value` 不能混用两种驱动模式。

## 4. 🆕 登录系统 (v1.3 重新设计)

### 4.1 登录方式
| 方式 | 说明 |
|------|------|
| **验证码登录（主推）** | 输入邮箱 → 发6位OTP → 验证。新用户和重置密码均走此流程 |
| **密码登录** | 邮箱+密码。仅限已设密码的用户 |
| **快捷登录** | 已登录同号 → 直接进；不同号 → Alert 弹窗让用户选方式，**不发验证码** |
| **Face ID** | 设置页「安全」分区独立开关，开启后冷启动自动扫脸 |

### 4.2 注册流程
```
验证码登录 → 首次用户 → 弹窗「设置密码」→ 可跳过
→ 设置密码(确认) → 设置用户名(12中文/24英文上限) → 完成
```

### 4.3 安全信息
- 设置页显示「📧 邮箱已验证 ✅」
- 「🔐 密码保护」→ 已设密码显示「✅ 修改」点击可修改密码；未设密码显示「⚠️ 点击添加」
- 忘记密码：密码登录页底部「忘记密码」→ 发验证码 → 验证 → 直接设新密码

### 4.4 hasPassword 检测机制
三道防线：① `SecureStore` 本地 `gt100_has_pw_{uid}` 标记 ② `UserProfile.hasPassword` 缓存 ③ Supabase `profiles.has_password` 表

### 4.5 多账号
- `SecureStore` 存 `gt100_saved_accounts` (最多5个)，含 email/nickname/avatarEmoji/lastLogin
- 支持删除账号（二次确认）
- 密码输入框配置 `textContentType` / `autoComplete` 属性支持 iOS 密码自动填充

## 5. 🆕 双人模式 (Couple Mode)

### 5.1 技术架构
`src/services/couple.ts` 提供完整双人服务：
- **邀请码**：`generateInviteCode()` 生成长串 hex（如 `a3f2b901-d4e5-7c2a`），碰撞概率≈0，24h过期
- **结对绑定**：`createInvite` / `claimInvite` → Supabase `couples` 表
- **共享清单**：`pushSharedList` / `pushSharedItems` → Supabase `shared_lists` / `shared_items` 表
- **状态同步**：`pushItemStatusChange` + Realtime `subscribeSharedItems`
- **回忆墙**：`pushMemory` / `fetchMemories` / `subscribeMemories` → Supabase `shared_memories` 表

### 5.2 Realtime 实时通知
`ListDetailScreen` 订阅 Supabase Realtime channel，伴侣完成/删除胶囊时：
- 即时刷新本地列表
- 粉色毛玻璃 Toast：「💌 Ta 刚完成了「xxx」」+ 触觉反馈
- 4s 自动消失

### 5.3 伴侣 UI
- 首页「💕 Ta 的共享清单」粉色边框卡片区
- 详情页右侧「回忆」按钮 → `SharedTimelineScreen` 时间线
- 长按伴侣清单卡片 → 直接进回忆墙

## 6. Supabase 表结构

### 需手动建表
```sql
-- invites 邀请码表
CREATE TABLE invites (id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, from_uid TEXT NOT NULL, code TEXT NOT NULL, expires_at TIMESTAMPTZ NOT NULL, claimed_by TEXT);

-- couples 结对关系表
CREATE TABLE couples (id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, partner1_uid TEXT NOT NULL, partner2_uid TEXT NOT NULL, status TEXT DEFAULT 'active');

-- shared_lists / shared_items / shared_memories (见 couple.ts 注释)

-- profiles 用户资料表
CREATE TABLE profiles (user_id TEXT PRIMARY KEY, nickname TEXT DEFAULT '', email TEXT DEFAULT '', avatar_emoji TEXT DEFAULT '👤', avatar_url TEXT DEFAULT '', has_password BOOLEAN DEFAULT false, updated_at TIMESTAMPTZ DEFAULT NOW());

-- Realtime
ALTER PUBLICATION supabase_realtime ADD TABLE shared_items;
ALTER PUBLICATION supabase_realtime ADD TABLE shared_memories;
```

### Supabase 后台配置
- Authentication → Settings → **关闭 Confirm email**（App 端 OTP 验证代替）
- Email Templates → 确保含 `{{ .Token }}`

## 7. 已知问题 & 待解决

| 1 | **profiles 表依赖** | 当前 `hasPassword` 检测优先用 SecureStore 本地标记 + 缓存，Supabase profiles 表查询仅做最后兜底。若 profiles 表未建，设置密码/用户名的数据无法云端持久化，仅存本地 |

| 6 | **SecureStore 超 2048 字节** | Supabase session token 改回 AsyncStorage 存储（`supabase.ts` 已配置），SecureStore 仅用于小数据 |
| 7 | **Face ID 自动登录依赖历史账号** | 冷启动扫脸后自动读取 `savedAccounts[0]` 发 OTP，若账号列表为空则不触发 |
